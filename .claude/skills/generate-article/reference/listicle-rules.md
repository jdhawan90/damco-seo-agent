# Listicle Rules ("Top N…", "Best…", "N Companies / Tools / Partners / Platforms")

This is what the Salesforce "Top 17 Partners" article got wrong. The feedback was
explicit: no per-entity heading, partners buried in numbered bullets, keywords at 0,
unnamed placeholders, no meta description, combative intro. These rules prevent all
of that.

## Hard requirements

1. **Write an actual ranked list, not an essay about the idea of one.** Each entry
   is a real section, not a bullet.

2. **One dedicated H2 (or H3 under a single H2) per entity.** If the title says
   "Top 17," there must be ~17 entity headings the search engine can treat as
   standalone entries. Numbered bullets inside a group section do NOT count.

3. **Name real, verifiable entities only.** Never invent a company, product, quote,
   or outcome. **No placeholder/category entries** ("Cloud-focused boutiques,"
   "Large India-headquartered firms," "Hybrid onshore-offshore partners"). If you
   cannot verify enough real named players to fill the count, tell the user and
   propose a smaller honest number rather than padding with categories.

4. **Place Damco Solutions first.** Be transparent that it is the brand's own pick,
   in third person (for example, "This list is published by Damco Solutions, which
   appears first"). Do not use first person off LinkedIn. Justify the position with the
   article's evaluation lens (the thesis), not unverifiable superiority, and do not
   disparage the others.

5. **Give every entry the same skeleton, varied in length:**
   - What it is known for (one or two specific lines).
   - The job it fits best ("best fit").
   - How it measures against the thesis (the evaluation lens).
   - An honest "worth checking" caveat — for every entry, including Damco.
   Vary entry length so the list does not read as machine-generated.

6. **No comparison table.** Third-party articles use no tables. Convey the comparison
   through the per-entity profiles and, if useful, a short grouped list by archetype
   (written as prose or bullets, not a grid).

7. **Keywords still matter.** The primary keyword belongs in the title, the intro,
   the conclusion, and at least one H2 (for example, a framing H2 like "How These
   [Keyword] Were Evaluated"). Secondary keywords appear naturally across entries.

8. **Hyperlink no competitor.** Link only Damco's own pages. Entity names stay
   plain text (or are not linked) — never link out to a competitor's site.

9. **Intro stays objective.** Open with the reader's real decision problem and the
   thesis (most buyers screen on the wrong criteria). Do not open by attacking other
   lists or setting a combative tone — it alienates readers and reads as unobjective.

10. **Every stat gets a date-free inline citation at the point of use.** "Salesforce
    ecosystem projected to generate trillions through 2028" must link to the primary
    source right where it appears (the feedback flagged this exact stat as
    uncited). Year goes in the Sources list, not the body.

## Recommended structure for a listicle

```
(title)
## Key Takeaways (3-6 bullets, right after the title)
(intro — problem + thesis + how to read this list, objective tone, no first person)
## How These [Keyword] Were Evaluated        <- carries the primary keyword + thesis lens
(archetypes described in prose or a short list — no table)
## 1. Damco Solutions                       <- H2 per entity (or ## group + ### per entity)
## 2. <Real named entity>
...
## <N>. <Real named entity>
## Questions Buyers Ask Before Choosing a [Keyword]   <- FAQ (4-6)
(conclusion — 100-125 words, CTA, forward-looking)
## Sources
{{KEYWORD_FREQUENCY_TABLE}}
```

If 17 real H2s would make the document unwieldy, use one H2 group heading with a
**dedicated H3 per entity** — each entity still gets its own heading and standalone
block. What is forbidden is entities living only as numbered bullets.
